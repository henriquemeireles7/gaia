## Problem Statement

A new class of builder is being created in real time by the AI-coding moment: no-code/low-code refugees graduating from Lovable, Bolt, V0, Make, and n8n into Claude Code. They have hit the no-code ceiling, want real code with production-grade infrastructure, but don't have the senior-developer instinct to assemble 6–10 vendors (Vercel + Supabase + Sentry + PostHog + Inngest + Resend + Linear + Mintlify + admin tooling) and stitch them into a working SaaS. Existing tooling does not serve this segment. Vercel and Supabase target classical developers who want full control and ejectability. Lovable, Bolt, V0, and Replit Agent target non-coders inside sandboxed environments with vendor lock-in and a hard ceiling. Nobody is targeting the **graduating segment** — builders ready to write real code today, with AI as collaborator, who need a complete production-grade stack out of the box.

The result: this segment spends weeks on plumbing, hits the same ceiling repeatedly, or never ships. The opportunity is to build the integrated stack — code, methodology, runtime, services — that makes "first SaaS deployed" a same-day proposition for the graduating builder.

## Demand Evidence

**What exists today:**

- Founder is the ICP. Power-user history with Make/n8n, hit the no-code ceiling, hated Lovable, moved to Claude Code as a first-time real-code builder. Has shipped 10+ personal projects in this paradigm and accumulated lived pain across distribution, security, performance, observability, and the "factory-trap" of over-engineering rather than shipping.
- Continuous observation: months of daily X feed reading documenting the no-code-to-real-code migration in real time.
- Qualitative pull: when founder describes their setup to peers, they consistently ask "how are you shipping so fast?" That signal is for the _workflow_, not yet for Gaia-as-a-product. The bridge from one to the other is the launch.

**Honest gap:**

- Zero paying customers, zero signed LOIs, zero users panicking when something breaks today.
- Zero specific human (other than the founder) named as a future user with a specific use case.

**Validation strategy:**
**Ship + measure**, not interview-driven validation. Justified by the paradigm-shift frame: the segment doesn't yet exist as a clearly articulable group, so interviews with current-paradigm users would be misleading. Henry Ford / Stripe pre-payments / Notion are the lineage. The risk is that this posture becomes confirmation-bias armor; the discipline that makes it work is **post-launch ship-velocity** — first-dollar within 60 days of paid launch, sunset features that don't convert.

## Status Quo

The graduating builder today picks one of two unsatisfying paths:

**Path 1 — Stack assembly.** Vercel ($20–40) + Supabase or Neon ($25–69) + Sentry ($26+) + PostHog (free–$$) + Inngest ($0–50) + Resend ($0–20) + Linear ($8–16) + Mintlify ($120) = ~$230–400/mo across 6–10 vendor relationships, each with its own dashboard, billing, integration, and auth model. Setup time: weeks of plumbing before a single feature ships.

**Path 2 — No-code platform.** Lovable ($20–50/mo), Bolt, V0, Replit Agent. Faster initial ship, but ceilings out. Vendor lock-in is real — you cannot extract your code and run it elsewhere when you outgrow the platform. Internal tooling needs (CRM, project tracking, docs) are usually outside scope.

### Direct competitors (integrated SaaS starter kits)

These are the actual head-to-head competitors at OSS launch — not Vercel/Supabase, who occupy a different layer:

**The third option that doesn't exist yet:** an OSS, agent-native, methodology-bundled stack that gets a graduating no-code builder to a deployed SaaS in one same-day session, with production-grade infrastructure, ejectable later. That's Gaia's wedge.

## Constraints

- Solo founder + AI as collaborator. No team to delegate to.
- TypeScript-only, Bun-only stack. Locked in vision v6 (§Stack).
- MIT licensed for OSS template + harness + methodology + skills + d-\* skills + provider-files architecture.
- "Boil 20% of every lake" discipline, made operational: **for any tool we replace, ship only the single feature its median user uses; never the second-most-common feature.** If a request requires the second feature, send the user to the original tool with a one-line config swap.
- Provider-files architecture from day 1. Every external dependency is a swappable provider file. Migration is a config-line change.
- Locked decisions in vision v6 stand: TypeBox, Drizzle, Elysia, SolidStart, Bun, Polar, Better Auth, etc.
- Self-evolving harness deferred to v2 (per vision §Harness). v1 ships static enforcement.

### Claim and trademark hygiene (added in spec review round 4)

Marketing copy and external-facing docs must NOT contain:

- "X-replacement" framing where X is a named live competitor (Mintlify, Linear, Sentry, PostHog, etc.). These names appear in _internal premises and risk wording only_, never in launch copy, README hero, landing page, X thread, Show HN post, or migration guides.
- Unverifiable guarantees: "production-grade", "same-day to deploy", "ejectable later" used as guarantees rather than design intents. Replace with falsifiable language: "wired with auth + payments + DB out of the box" (verifiable), "Railway deploy config included" (verifiable), "swap-ready provider files for payment/email/observability" (verifiable, see Premise #9).
- Trademark exposure: register a trademark/IP-review pass to the pre-launch checklist (must-ship Stage 1 v0.1) — confirm Gaia name does not conflict, and the comparison framing in guides ("Migrate from Lovable") cannot be construed as competitor disparagement.

Internal premise wording can keep the original comparisons since premises are not public assets.

## Risks

Material risks the play has to survive (priority-ordered after spec review round 4 cross-model agreement: security/claim and legal exposure outrank support load and channel miss):

1. **Security incident or claim mismatch.** Shipping auth + payments + webhook handling that contains a CVE-level defect, or shipping marketing claims ("production-grade", "same-day deploy") that the codebase cannot honor at launch. Either burns trust with the Lovable-graduate ICP for years, since they are risk-averse and watching for "is this real code?" signals. **Mitigation:** 24-hour pre-launch CVE/dependency audit on Polar + Better Auth + Bun + Drizzle (lift to must-ship Stage 1 v0.1); strip unverifiable claims from launch copy (see Claim and trademark hygiene); ship `.env.example` (no real values) + gitleaks pre-commit hook in the _template_ (not just in our repo); document key-rotation policy in README.
2. **Vendor fragility on early-stage stack.** Better Auth, Polar, SolidStart, Bun are all <3 years old. Any one deprecating/pivoting forces a Gaia-side rewrite. **Mitigation:** provider-files architecture for payment/email/observability/storage (see Premise #9 — swappable as config-line changes). Runtime/framework/auth swaps are documented in README as architectural rewrites, not config swaps; do not promise otherwise. Pin exact versions in package.json. Acceptance test for provider files lands in Stage 1 v0.1 (Stripe stub + CI swap test).
3. **ICP misread.** If 60 days post-Stage-1 has <50 stars and zero engagement, or if Premise #11's Day-60 falsifier triggers (no-code-to-real-code candidates can't be triangulated above 100/month), the Lovable-graduate segment may not exist as a coherent group yet. **Mitigation:** abandonment ladder in Success Criteria; pre-committed external accountability (named friend or weekly public X post with stars/clones/DMs/falsifier metrics) so the same biased founder isn't sole reviewer of negative signals.
4. **Legal and trademark exposure.** "X-replacement" framing in launch copy invites C&D. Trademark conflict with "Gaia" name unverified. "Migration from Lovable" guides could be read as competitor disparagement under some jurisdictions. **Mitigation:** trademark/IP-review pass added to must-ship Stage 1 v0.1 (see Claim and trademark hygiene). Strip all "X-replacement" from public-facing copy. Confirm Gaia trademark availability before any branded launch artifacts.
5. **Support-burden risk if Stage 1 succeeds.** Solo OSS maintainer with 500+ stars typically receives 30–60 issues/PRs/week. **Mitigation:** opinionated CONTRIBUTING.md, automated triage hooks, willingness to close issues that don't fit the methodology. Note: this risk is over-emphasized historically; first 90 days rarely see this volume. Recalibrated as #5, not #1.
6. **Confirmation-bias armor.** Founder has explicitly pushed back on user-validation framing. The same instinct that defends a paradigm-shift bet can blind to contradictory data post-launch. **Mitigation:** pre-committed external accountability hook (see Risk #3) — same mitigation, doubles up because both risks share the failure mode.
7. **Show HN front-page miss.** Solo OSS launches without an existing audience hit Show HN front page ~10–15% of the time. **Mitigation:** primary launch channel is X/Twitter thread + targeted DMs; Show HN is a bonus, not the strategy. Decide Show HN day-1 or day-21 explicitly when scheduling, not "week 3 or 4."
8. **Reference-file scope creep (factory-trap residual).** The doc dependency "reference files complete enough to teach the methodology" is fuzzy and will expand silently. **Mitigation:** **hard cap at 6 reference files for Stage 1, each ≤500 lines, each ends in 3 worked examples.** Remaining 6 deferred to Stage 1.5. Anything beyond 12 total is a memo or rejected. Note: 17 reference files already exist on disk in `.gaia/reference/`; Stage 1 work is to verify the cap (≤500 lines, 3 examples each) on the chosen 6 and trim/defer the rest, not write from scratch.
9. **Admin-panel scope creep.** Pre-built admin (metrics + health + email + mini-CRM + projects + docs) is genuine 6-tool replacement scope for a solo founder. v0.1 must be ruthlessly capped. **Mitigation:** strict v0.1 cap at admin scaffold + 2 modules (metrics + email log). v0.2–v0.5 each add 1–2 modules with explicit scope and a ship date. No "one more module before launch" — modules ship after launch, in version increments, on the live repo. Per-module sunset criteria added: any module with <10% of weekly-active admin users opening it within 60 days post-launch is killed.

### Skill usage examples (Stage 1 onward)

`/d-admin` is a Claude Code skill (markdown-only, no CLI). The user invokes it inside a Claude Code session:

```
# In Claude Code, after shipping a `customer_interviews` feature with a Drizzle schema:
/d-admin add customer_interviews
# The skill reads the schema from packages/db/schema.ts, generates list + detail + edit views,
# wires them into apps/web/src/routes/admin/, and opens the new admin URL.
```

A separate `/d-cms` (or extending `/d-content`) handles content publishing — also a skill, no CLI.

The pre-built admin panel modules (metrics, health, email log, mini-CRM, projects, docs) are NOT generated by these skills — they ship pre-built in the OSS template. The skills exist for the _user's own_ features that don't exist in the template.

**Note (round 4 reality-match):** there is no `bun gaia` CLI. Earlier doc revisions referenced one; that was inaccurate. Stage 1 v0.1 ships only the markdown skill files (`.claude/skills/d-admin/SKILL.md` plus content extension), not a CLI binary.
