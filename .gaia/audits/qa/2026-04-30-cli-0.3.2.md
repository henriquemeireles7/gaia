## ISSUE-001 — `bun dev` returns HTTP 503 on every route (CRITICAL)

**Severity:** Critical · **Category:** Functional (SSR) · **Reproduces in:** fresh scaffold AND source repo on `v032-review-fixes`

**Repro:**
```bash
cd /tmp && rm -rf gaia-qa-032 && mkdir gaia-qa-032 && cd gaia-qa-032
bun create gaia-app@latest myapp
cd myapp && bun dev &
sleep 8
curl -s -o /dev/null -w "%{http_code}\n" http://localhost:3000/        # → 503
curl -s -o /dev/null -w "%{http_code}\n" http://localhost:3000/login   # → 503
curl -s -o /dev/null -w "%{http_code}\n" http://localhost:3000/dashboard  # → 503
```

**Server response body:**
```
Error: Client-only API called on the server side. Run client-only code in
onMount, or conditionally run client-only component with <Show>.
  at notSup (solid-js@1.9.12/web/dist/server.js:760:9)
  at @solidjs/router@0.15.4/dist/index.js:1641:27
  at apps/web/src/routes/dashboard.tsx?pick=route:5:31
  at apps/web/src/entry-server.tsx:8
```

**Root cause (best read):** `@solidjs/router@0.15.4`'s precompiled `dist/index.js` calls `template(\`<a>\`)` at module top-level. On the SSR side, `template()` is the `notSup` stub, so any SSR module-graph load that touches the router throws. The router's `package.json` declares an `exports."." = { "solid": "./dist/index.jsx", "default": "./dist/index.js" }` map — vinxi/SolidStart 1.3.2's SSR resolver is reading `default` instead of `solid`, so it never goes through `vite-plugin-solid` to compile the JSX. `apps/web/app.config.ts` has no plugin overrides and no `conditions: ['solid']` — that's the missing lever.

**Same pins resolve in source repo:** I confirmed `bun dev` from `/Users/henriquemeireles/.../gaia/west-monroe` returns the same 503. So this is not a 0.3.2 regression — `master`'s state ships verbatim into the published template, and 0.3.1/0.3.2 QA passes never invoked `bun dev` end-to-end.

**Suspected fix paths (need investigation, not blind apply):**
1. Pin `@solidjs/router` to a known-good version where SSR `dist/index.js` doesn't top-level `template()` (probably ≤0.14.x or wait for upstream fix).
2. Add the `solid` resolution condition to vinxi's SSR Vite config in `apps/web/app.config.ts` so `dist/index.jsx` is loaded and compiled.
3. Add a `bun dev` smoke test to `bun run check` so future template drift is caught at PR time.

**Files to look at:**
- `apps/web/app.config.ts` (current: 7 lines, no plugin overrides)
- `apps/web/src/entry-server.tsx:8`
- `node_modules/@solidjs/router/package.json` exports map

**Why this is a P0 for 0.3.2:** the README's "Quick start" reads:
```
bun create gaia-app@latest myapp
cd myapp
bun dev    # mock mode — see your app at http://localhost:3000
```
The first thing a new user does after scaffold is `bun dev`. They'll see a 503 stack trace in their browser within 60 seconds of installing the package.

## ISSUE-002 — `bun gaia <verb>` is unreachable after scaffold (CRITICAL)

**Severity:** Critical · **Category:** Functional (CLI) · **Reproduces in:** fresh scaffold AND source repo

**Repro:**
```bash
cd /tmp/gaia-qa-032/myapp
bun gaia status         # → error: Script not found "gaia"
bun gaia explain E1001  # → error: Script not found "gaia"
bun gaia --help         # → error: Script not found "gaia"
ls node_modules/.bin/gaia   # → No such file or directory
bunx gaia --help        # → error: could not determine executable to run for package gaia
                        #   (also tries to install an unrelated `gaia` package off npm — gnarly)
```

**Root cause:** `cli/package.json` declares two bins under the `create-gaia-app` package:
```json
"bin": {
  "create-gaia-app": "./src/create.ts",
  "gaia": "./src/cli.ts"
}
```
`bun create gaia-app@latest <name>` runs the `create-gaia-app` bin once and is done. The scaffolded project does **not** depend on `create-gaia-app`, so:
- `node_modules/.bin/gaia` never gets symlinked into the project
- the scaffold's `package.json` has no `"gaia"` script
- `bun gaia` fails the script-then-bin lookup
- `bunx gaia` collides with an unrelated `gaia` package on npm (and, worse, that package's postinstall starts compiling grpc — kill it fast)

The README confidently says:

> `bun gaia` resolves via the scaffolded project's `node_modules/.bin/`

This is not how it works for the published artifact. The README is describing how it works inside the *source repo* (where `cli/` is a workspace package and Bun symlinks the bin from a sibling workspace). That linkage doesn't survive packaging to npm.

**Suspected fix paths:**
1. Add `create-gaia-app` to the scaffolded project's `devDependencies` and `bun install` it during scaffold (already running install, so cost is ~0). Then `node_modules/.bin/gaia` exists.
2. Add a `"gaia": "bunx --package create-gaia-app gaia"` script to the scaffolded `package.json`.
3. Write a tiny `gaia` shim into the scaffolded project's root or `node_modules/.bin/` during scaffold.

Option 1 is the most idiomatic. It also gives users `bun update create-gaia-app` to upgrade the verb runner alongside their template.

**Why this is a P0:** every README on-ramp after the first one (`bun gaia live`, `bun gaia deploy`, `bun gaia smoke`, `bun gaia verify-keys`) is currently dead.

## ISSUE-003 — `bunx gaia` triggers an unrelated package install with native compilation (HIGH, side-effect of #002)

**Severity:** High (developer-trust impact) · **Category:** UX

When a user follows the broken-but-plausible path of `bunx gaia status` (reasonable assumption when `bun gaia` says "Script not found"), they hit a different `gaia` package on npm. That package has a `grpc` dependency whose postinstall runs `node-gyp` against C++ sources for ~30+ seconds before any error surfaces. I had to `pkill -f node-gyp` to stop it during this audit.

A user without context will sit watching a foreign package compile native code from their CLI typo. That's a trust event the scaffolder should not enable.

**Mitigation:** fixing #002 closes this — once `node_modules/.bin/gaia` exists, `bunx` is no longer the natural fallback.
